## Role-Specific Communication Guide

### Executive Sponsors & Leadership

**What They Need to Know:**
- Business impact and ROI
- Timeline and milestones (on track or at risk?)
- Budget and resource status
- Major risks and escalations
- Success metrics and outcomes

**Communication Cadence:** Monthly or quarterly  
**Format:** Executive summary (1 page), live meetings  
**Tone:** Strategic, outcome-focused, risk-transparent

**Sample Message:**
```
Project Status Summary - [Project Name]

Progress: [Milestone % complete]
Timeline: On track / At risk (reason)
Budget: [% spent of allocated]
Key Wins: [2-3 recent accomplishments]
Risks: [Top 1-2 risks and mitigation]
Next Steps: [Key milestones for next month]
Decision Needed: [Any escalations or approvals]
```

---

### Product & Business Stakeholders

**What They Need to Know:**
- Feature progress and delivery timeline
- Customer feedback and market feedback
- Quality and testing status
- Dependencies and risks
- Success metrics and adoption

**Communication Cadence:** Bi-weekly or as-needed  
**Format:** Email updates, sprint reviews, demos  
**Tone:** Data-driven, customer-focused, collaborative

**Sample Message:**
```
Feature Update - [Feature Name]

Current Status: [Planning / In Development / Testing / Done]
Timeline: Expected release [date]
Customer Feedback: [Key customer insights or feedback]
Blockers: [Any dependencies or risks]
Next Milestone: [What's coming next]
Metrics to Watch: [Key success indicators]
```

---

### Development & Technical Teams

**What They Need to Know:**
- User stories and acceptance criteria (from BA, Product)
- Design specifications and requirements (from UX, BA)
- Technical dependencies and risks (from Tech Lead, Project Manager)
- Quality and testing expectations (from QA Lead)
- Security requirements and constraints (from Security Champion)

**Communication Cadence:** Daily standups, sprint planning  
**Format:** User stories, design specs, technical documentation, daily sync  
**Tone:** Clear, detailed, collaborative, unblocking-focused

**Sample Standup:**
```
What Did We Build Yesterday?
- [Feature work, PRs merged, tests passed]

What Are We Building Today?
- [Planned work, dependencies to coordinate]

What's Blocking Us?
- [Blockers, questions for BA/Design/QA/Security]
```

---

### Quality Assurance & Testing Teams

**What They Need to Know:**
- Acceptance criteria and Definition of Done
- Test strategy and test types needed
- Feature readiness for testing
- Quality metrics and coverage targets
- Release readiness criteria

**Communication Cadence:** Sprint planning, daily (if testing in progress), pre-release  
**Format:** User stories with AC, test plans, quality dashboards, release checklists  
**Tone:** Detailed, specific, quality-focused, collaborative

**Sample Message:**
```
Ready for QA - [Feature Name]

Acceptance Criteria:
- [Testable AC 1]
- [Testable AC 2]

Test Types Required:
- [Unit, Integration, E2E, Performance]

Known Issues:
- [Any known limitations or workarounds]

Timeline:
- Ready: [Date]
- QA Complete: [Target date]
```

---

### Customer Support & Operations

**What They Need to Know:**
- Feature descriptions and customer-facing benefits
- Launch date and rollout plan
- Known issues and workarounds
- Customer communication messaging
- Support documentation and training needs

**Communication Cadence:** Pre-release, release week, post-release  
**Format:** Email briefings, documentation, training sessions  
**Tone:** Customer-focused, clear, action-oriented

**Sample Message:**
```
Feature Release Brief - [Feature Name]

What: [Customer-facing description]
Why: [Customer benefit or problem solved]
When: [Release date and rollout approach]
Impact: [Changes to existing workflows or processes]
Training: [Support materials and training available]
FAQ: [Anticipated customer questions]
Escalation: [Who to contact if issues arise]
```

---

### Security & Compliance Teams

**What They Need to Know:**
- Security requirements and threat models
- Data handling and compliance implications
- Code review findings and remediation
- Dependency updates and vulnerability scans
- Release approval status

**Communication Cadence:** Design phase, code review, pre-release  
**Format:** Threat models, security requirements docs, code review comments, compliance checklists  
**Tone:** Risk-focused, technical, collaborative

**Sample Message:**
```
Security Review Complete - [Feature Name]

Threat Model: [Summary of analysis]
Issues Found: [Critical / High / Medium / Low]
Remediation: [Fixes applied, due date]
Compliance Impact: [GDPR / SOC2 / etc.]
Approval: [Approved / Conditional]
Next Steps: [Dependencies, follow-up items]
```

## Tips for Effective Stakeholder Communication

1. **Know Your Audience:** Tailor message content, format, and technical level to stakeholder role
2. **Be Transparent:** Share risks, blockers, and challenges proactively—not just wins
3. **Focus on Impact:** Connect project progress to business/customer outcomes
4. **Be Consistent:** Use templates and regular cadence so stakeholders know what to expect
5. **Listen:** Incorporate stakeholder feedback and concerns into project decisions
6. **Use Data:** Support claims with metrics, trends, and evidence
7. **Escalate Appropriately:** Flag high-priority issues to relevant stakeholders quickly
8. **Document Decisions:** Record decisions and rationale for future reference
